# redbytefx v0.2 status

## What `v0.2` means

`v0.2` is the shader-stdlib milestone.

The goal is not "publish the library" and not "finish the product". The goal is:

- keep `core` and `compose` small and understandable
- build a compact but strong standard library of shader primitives
- add a first wave of recipe-level helpers that prove the DSL can scale beyond toy snippets

## Current assessment

Current level: `late v0.2`, with some `v0.3 Tooling` work already visible in the sample app.

Why this is not just mid-`v0.2` anymore:

- `:redbytefx-core` and `:redbytefx-compose` already support real live effects, runtime bindings, and generated AGSL inspection
- `:redbytefx-stdlib` is no longer hypothetical; it already spans procedural, color, masks, timing, compositing, transitions, gradients, signal, polar, lighting, SDF, and routing
- the sample app proves the helpers in a real catalog instead of isolated ad-hoc previews

Why this is not fully "done v0.2" yet:

- the stdlib surface is strong, but still broader and noisier than a truly curated first milestone should be
- naming, defaults, and helper boundaries still need one deliberate pass
- docs explain the system reasonably well, but not yet at the level where authoring feels guided and obvious
- some of the strongest current progress is already tooling-oriented and belongs conceptually to `v0.3`, not to the definition of `v0.2`
- some early `v0.6` shaping is now visible in package docs and sample guidance, but that does not replace the remaining stdlib curation work

## What is already in good shape

- low-level DSL authoring in `:redbytefx-core`
- Compose integration in `:redbytefx-compose`
- a substantial stdlib surface in `:redbytefx-stdlib`
- package-level guidance for the canonical `core` vs `stdlib` split and parameter vocabulary
- aligned KotlinDoc on the canonical starter helpers so coordinate space and “start here” families are easier to teach consistently
- generated AGSL inspection
- automated quality gate via `./gradlew qualityCheck`
- compiler/runtime unit tests plus restored AGSL golden snapshots
- representative stdlib golden snapshots for coordinate and masking pipelines
- representative compiler coverage across the main stdlib helper families

## What still needs to happen to truly finish `v0.2`

### 1. Stdlib curation pass

- review helper names across all stdlib files
- keep normalizing defaults, parameter ordering, and parameter vocabulary
- decide which helpers are canonical and which are still experimental/noisy
- reduce accidental overlap between helpers that solve nearly the same problem

Current direction:

- treat `normalizedUv()`, `sampleUv(...)`, `aspectCenteredUv(...)`, `circleMask(...)`, `rectMask(...)`, `maskedMix(...)`, `softFill(...)`, `softStroke(...)`, `segmentMask(...)`, and `segmentProgress(...)` as the main starter path
- keep broader style helpers like `fbm(...)`, `domainWarp(...)`, `chromaticOffset(...)`, `frameMask(...)`, and `cornerMask(...)` available without letting them crowd the first mental model

### 2. Boundary cleanup

- make the `core` vs `stdlib` split more explicit in docs and KotlinDoc
- confirm that recipe-level helpers do not leak back into `core`
- make the intended mental model of each module obvious

### 3. Documentation pass for the stdlib milestone

- add stronger module-level guidance in README
- document the most important helper families, not just list them
- explain what `v0.2` deliberately does not solve yet

### 4. Test coverage tightening

- keep the AGSL golden tests healthy
- ensure each major helper family has representative compiler coverage
- treat regressions in generated source shape as important, not incidental

## Exit criteria for "done v0.2"

`v0.2` should be considered finished when:

- the stdlib feels curated rather than merely broad
- helper names feel deliberate and teachable
- `core` and `stdlib` responsibilities are clear
- the main helper families are documented and covered by tests
- adding more helpers stops being the default answer to every missing pattern

## Important note

The project already contains early `v0.3 Tooling` work in the sample app. That is good progress, but it should not hide the fact that the library surface itself still needs one final shaping pass before `v0.2` is truly settled.
