# v0.6 Library shaping

**Goal:** a clear library identity **before** any Maven/publishing conversation: canonical vs exploratory surface, examples, and honest limits.

## What is “done” for this documentation milestone

The following are **in tree** and treated as the **first complete v0.6 guidance pass** (not a frozen public API contract):

- **Canonical vs exploratory** is explicit in `:redbytefx-stdlib` package KDoc (`RedByteFxStdlib.kt`), README family map, and sample **PATH** signals (`START HERE`, `CANONICAL`, `EXPLORATORY`, `FOUNDATION`).
- [cookbook-patterns.md](cookbook-patterns.md) and [agsl-vs-redbytefx.md](agsl-vs-redbytefx.md) align with the same teaching order.
- Exploratory helpers remain available but are framed as **advanced** relative to the canonical spine.
- **[library-limitations.md](library-limitations.md)** — single index for platform/runtime limits, sampling-space expectations, and tooling/API caveats (links out to detailed notes).

## What remains incremental

- Naming defaults and parameter order on specific stdlib helpers as usage grows.
- Tighter “one obvious way” for a few noisy families — only when real confusion shows up.
- Publication and semver — **out of scope** until explicitly chosen (see [roadmap.md](roadmap.md)).
